# catch_fish 代码阅读顺序

> 从零开始复现这个项目的最佳阅读路径。由底层到上层，由简单到复杂。

---

## 阅读总纲

```
第 1 轮: 基础层（配置、工具、数据模型）          → 理解"数据长什么样"
第 2 轮: Agent 层（基类 + 三个子 Agent）         → 理解"谁干什么活"
第 3 轮: 协议层（MCP Server、Orchestrator）      → 理解"怎么协作"
第 4 轮: 对话层（Session、ChatAgent）            → 理解"怎么多轮聊天"
第 5 轮: 接入层（Gateway、API、部署）            → 理解"怎么对外服务"
```

---

## 第 1 轮：基础层（3 个文件，约 30 分钟）

这是整个项目的地基，不依赖任何上层模块。

### 1.1 `src/config.py`

**为什么先读它**：所有模块都从这里拿配置，先搞清楚有哪些开关。

| 关注点 | 说明 |
|--------|------|
| `Settings` 类 | 继承 `pydantic_settings.BaseSettings`，自动从 `.env` 读配置 |
| `model_config` | `env_file=".env"` 指定了配置文件路径 |
| 关键字段 | `deepseek_api_key`、`database_url_async`、`xianyu_cookie` |
| `is_development` | 环境判断属性，影响中间件行为 |

**复现要点**：理解 `pydantic-settings` 的用法——你只需要定义字段名和类型，它会自动匹配环境变量。

---

### 1.2 `src/utils/logger.py`

**为什么第二个读它**：其他模块第一行就是 `get_logger(__name__)`。

| 关注点 | 说明 |
|--------|------|
| `logger.remove()` | 先清除 loguru 默认 handler |
| 控制台输出 | 彩色格式，`{time} {level} {name}:{function}:{line} {message}` |
| 文件轮转 | 每天午夜轮转，保留 30 天，压缩为 zip |
| 错误单独记录 | ERROR 级别单独文件，保留 90 天 |

**复现要点**：`loguru` 比标准库 `logging` 简单很多——不需要配置 dict，直接 `logger.add()` 即可。

---

### 1.3 `src/models/schemas.py`

**为什么第三个读它**：定义了所有 Agent 之间传递的数据结构。读完这个，你就知道整个系统的"信息流"长什么样。

| 阅读顺序 | 类名 | 用途 |
|----------|------|------|
| 1 | `ParsedIntent` | Orchestrator 解析出的用户意图 |
| 2 | `XianyuItemOut` | 单条闲鱼商品 |
| 3 | `FinderResult` | Finder Agent 输出（包含商品列表） |
| 4 | `ChannelPrice` | 单个渠道的新品价格 |
| 5 | `EncyclopediaResult` | Encyclopedia Agent 输出（新品信息） |
| 6 | `Recommendation` | 单个推荐（含评分） |
| 7 | `MarketSummary` | 市场概况 |
| 8 | `CalculatorResult` | Calculator Agent 输出（完整分析） |
| 9 | `SearchResultResponse` | 最终返回给用户的完整结果 |
| 10 | `ChatRequest` / `ChatResponse` | 多轮对话的消息格式 |
| 11 | `SSEProgressEvent` | SSE 流式进度事件 |

**阅读技巧**：注意观察每个 Schema 的字段类型——比如 `FinderResult.items` 是 `list[XianyuItemOut]`，你就知道 Finder 的输出会被 Calculator 消费。

---

### 1.4 `.env.example`

快速浏览，和环境变量混个脸熟。重点关注 `DEEPSEEK_API_KEY` 和 `DATABASE_URL`。

## 第 2 轮：Agent 层（7 个文件，约 1 小时）

### 2.1 `src/agents/base.py` — Agent 基类

**整个 Agent 体系的基础**，定义了所有 Agent 的通用能力。

| 关注点 | 说明 |
|--------|------|
| `BaseAgent` 抽象类 | `agent_id`、`agent_name` 标识身份 |
| `client` 属性 | 延迟创建 `AsyncOpenAI`，避免启动时就连接 |
| `ask_llm()` | 发送消息给 LLM，带 `@retry` 自动重试（tenacity） |
| `ask_llm_json()` | 调 `ask_llm()` 然后解析 JSON——处理了 ` ```json ` 代码块 |
| `_parse_json()` | 静态方法：直接解析 → 从 markdown 提取 → 报错 |
| 抽象方法 | `execute()` 和 `system_prompt()` 子类必须实现 |

**复现要点**：
- `tenacity` 的 `@retry` 装饰器非常简单——`stop_after_attempt(3)` + `wait_exponential(min=1, max=30)` 就能实现指数退避重试。
- `_parse_json` 的容错逻辑值得学习：先 `json.loads`，失败了再找 ` ```json ` 标记。

---

### 2.2 `src/agents/finder/prompts.py` + `agent.py` — 商品搜索 Agent

**为什么先读 Finder**：它是三个子 Agent 里最简单的——核心逻辑就是"调 MCP 搜闲鱼 + LLM 整理数据"。

| 文件 | 关注点 |
|------|--------|
| `prompts.py` | `FINDER_SYSTEM_PROMPT` 定义了搜索策略（关键词精准化、价格异常检测、成色判断） |
| `agent.py` | `FinderAgent` 继承 `BaseAgent` |

**`agent.py` 的三个关键方法**：

```
execute()
  ├── _search_xianyu()      # 调 MCP 搜索
  ├── _normalize_results()  # LLM 整理结果（mcp 有数据时）
  └── _fallback_search()    # LLM 模拟数据（mcp 不可用时）
```

| 关注点 | 说明 |
|--------|------|
| 构造函数 | `__init__(self, mcp_client=None)` — MCP 客户端可选注入，方便测试 |
| `_search_xianyu()` | 调 `self.mcp.call_tool("search_items", ...)` — 注意 MCP 调用的参数结构 |
| `_fallback_search()` | 无 MCP 时让 LLM 生成模拟数据——开发调试的巧妙做法 |
| `_normalize_results()` | 用 LLM 清洗原始数据，保证输出结构一致 |

---

### 2.3 `src/agents/encyclopedia/prompts.py` + `agent.py` + `scrapers.py` — 商品百科 Agent

**核心逻辑**：并行抓取多个电商网站 → LLM 整合完善。

| 文件 | 关注点 |
|------|--------|
| `prompts.py` | 定义了数据获取优先级（官网 > 京东 > 天猫）|
| `scrapers.py` | `JDScraper` 和 `TmallScraper` 的具体抓取逻辑 |
| `agent.py` | `EncyclopediaAgent` 编排抓取 + LLM 研究 |

**`agent.py` 流程**：

```
execute()
  ├── _fetch_prices()       # 并行调 scrapers（目前是顺序循环，可优化为 asyncio.gather）
  ├── _research()           # LLM 整合抓取结果
  └── _find_lowest()        # 纯函数：找最低在售价
```

| 关注点 | 说明 |
|--------|------|
| `SCRAPERS` 字典 | `{"jd": JDScraper(), "tmall": TmallScraper()}`——注册模式，方便扩展 |
| `BaseScraper.parse_price()` | 正则提取价格：`r'[\d,]+(?:\.\d{2})?'` |
| `_research()` | 把抓取到的片段数据 + Prompt 模板丢给 LLM，让 LLM 补充缺失信息 |

---

### 2.4 `src/agents/calculator/prompts.py` + `agent.py` + `scoring.py` — 性价比计算 Agent

**三个子 Agent 中最复杂的一个**，有两层分析：规则引擎（确定性的）+ LLM（定性的）。

**必须先读 `scoring.py`**，它是整个项目**唯一不依赖 LLM 就能跑**的模块：

| 函数 | 逻辑 |
|------|------|
| `calc_discount_rate()` | `二手价 / 新品价` |
| `score_by_discount()` | 折扣率越低，分越高（满分 40） |
| `score_by_condition()` | 标准化成色 → 分数映射（满分 25） |
| `score_by_seller()` | 信用分段：≥800=15, ≥700=12, ... |
| `score_by_market()` | 供应量越多，议价空间越大（满分 10） |
| `calc_depreciation_factor()` | 距发布时间越久，折旧越大 |
| `calc_comprehensive_score()` | **主函数**：加权求和，5 个维度 |
| `normalize_condition()` | "99新" → `like_new`，"轻微划痕" → `good` |

**权重分配**：
```
折扣率 40% + 成色 25% + 卖家信用 15% + 市场供需 10% + 时间折旧 10% = 100%
```

**然后读 `agent.py`**：

```
execute()
  ├── 对每个闲鱼商品调 calc_comprehensive_score() → 规则评分
  ├── 按规则分数排序
  ├── _llm_analyze()        # LLM 做深度分析 + 写结论
  ├── 构建 best_deal         # 分数最高的作为最佳推荐
  ├── 构建 recommendations   # ≥60 分的前 5 个
  └── _determine_recommendation()  # 整体建议: buy_used / buy_new / consider
```

**复现要点**：规则引擎 + LLM 的混合模式是这个项目最重要的设计模式——规则引擎保证底线（不会给出离谱的评分），LLM 提供灵活性（自然语言结论、情景判断）。

### 3.2 `src/mcp/xianyu_server.py` — MCP Server

**当前是骨架**，工具返回占位数据。但架构完整：

| 关注点 | 说明 |
|--------|------|
| `XianyuMCPServer.__init__()` | 创建 `Server("xianyu-mcp-server")` + 注册 handler |
| `list_tools()` | 返回工具列表 |
| `call_tool()` | 根据 `tool_name` 路由到 `_handle_xxx()` |

### 3.3 `src/orchestrator/agent.py` — 编排器 Agent

| 关注点 | 说明 |
|--------|------|

| `parse_intent()` | 核心方法：自然语言 → `ParsedIntent` |

---

### 3.4 `src/orchestrator/workflow.py` — 工作流引擎

**这是整个系统最核心的编排逻辑**：

```
CatchFishWorkflow.execute()
  │
  ├── Step 1: _step_parse_intent()     ─── Orchestrator 解析
  │
  ├── Step 2: asyncio.gather(          ─── 并行
  │       _step_find(),                   Finder 搜闲鱼
  │       _step_research(),               Encyclopedia 查新品
  │   )
  │
  └── Step 3: _step_calculate()        ─── Calculator 算性价比
```

| 关注点 | 说明 |
|--------|------|
| `WorkflowContext` | dataclass，贯穿整个工作流的上下文对象 |
| `TaskState` | 每个子任务的状态（pending→running→done/failed） |
| `asyncio.gather` | **关键并行点**：Finder 和 Encyclopedia 无依赖，同时跑 |
| 错误处理 | 任一步骤抛异常 → `ctx.status = "failed"` → 向上传播 |

**复现要点**：理解为什么 Finder 和 Encyclopedia 可以并行——它们只依赖 `ParsedIntent`（来自 Step 1），互不依赖对方的输出。Calculator 必须等两者都完成。

## 第 4 轮：对话层（3 个文件，约 40 分钟）

这是多轮聊天能力的核心，第 1-3 轮是一次性搜索的基础，第 4 轮把它升级为持续性对话。

### 4.1 `src/gateway/session.py` — 会话管理器

| 关注点 | 说明 |
|--------|------|
| `ConversationMessage` | 单条消息：role + content + timestamp + metadata |
| `Session` | 一个完整会话：消息列表 + 搜索上下文 + 生命周期 |
| `session.add_message()` | 追加消息，更新 last_active |
| `session.bind_search_result()` | **关键方法**：把搜索结果绑定到会话，后续追问依赖它 |
| `session.get_item_by_index(3)` | 用户说"第三个"，返回对应商品 |
| `session.get_history_for_llm()` | 取最近 N 条消息转 LLM 格式 |
| `SessionManager` | 全局单例，create/get/delete/cleanup |

**复现要点**：
- `search_context` 是对话连续性的关键——一次搜索的结果可以被后续多轮追问引用
- `get_item_by_index()` 采用了 1-based 索引（用户说"第一个"就是 index=1）
- 过期清理策略：每次 `get()` 时检查 TTL

---

### 4.2 `src/agents/chat/prompts.py` — 对话 Prompt 模板

| Prompt | 用途 |
|--------|------|
| `CHAT_SYSTEM_PROMPT` | 定义人设：数码爱好者助手 |
| `INTENT_ROUTING_PROMPT` | 5 种意图的精确分类 |
| `FOLLOW_UP_PROMPT` | 追问模板——包含商品序号定位 |
| `COMPARE_PROMPT` | 多商品对比模板 |
| `GENERAL_CHAT_PROMPT` | 闲聊模板 |

**复现要点**：`INTENT_ROUTING_PROMPT` 中的 `{conversation_history}` 和 `{search_context}` 是意图路由准确的关键——LLM 需要知道"当前有什么可引用"才能正确判断 "第三个" 是追问还是新搜索。

---

### 4.3 `src/agents/chat/agent.py` — 对话 Agent

**整个系统的对话入口**，架构如下：

```
ChatAgent.handle_message(session, user_message)
  │
  ├── _route_intent()           ─── 判断意图（LLM + 规则辅助）
  │     ├── 首次消息？→ _looks_like_search() 快速判断
  │     └── 有上下文？→ _llm_route_intent() 精确判断
  │
  ├── _handle_new_search()      ─── 触发完整工作流 → 绑定结果到 session
  ├── _handle_follow_up()       ─── 从 session 取商品 → LLM 深度分析
  ├── _handle_compare()         ─── 取多个商品 → LLM 对比
  ├── _handle_detail()          ─── 取单个商品 → 详细解读
  └── _handle_general_chat()    ─── 直接 LLM 回复
```

| 关注点 | 说明 |
|--------|------|
| `_route_intent()` | 三层判断：首次消息 → 规则快速判断 → LLM 精确判断 |
| `_looks_like_search()` | 纯规则：检查是否包含搜索关键词 |
| `_llm_route_intent()` | 投喂对话历史 + 搜索上下文 → LLM 输出 intent JSON |
| `_handle_new_search()` | 调 `CatchFishWorkflow.execute()` → `session.bind_search_result()` |
| `_format_search_response()` | 把结构化 `SearchResultResponse` 格式化为自然语言 Markdown |

## 第 5 轮：接入层（5 个文件，约 30 分钟）

### 5.1 `src/gateway/middleware.py`

| 关注点 | 说明 |
|--------|------|
| `RequestLoggingMiddleware` | 记录每个请求的方法、路径、耗时 |
| `SimpleRateLimiter` | 内存限流器：滑动窗口，每分钟 N 次 |

**复现要点**：生产环境应把 `SimpleRateLimiter` 替换为 Redis 实现。

---

### 5.2 `src/gateway/router.py` — 一次性搜索 API

```
POST   /api/v1/search              # 发起搜索 → 202 + search_id
GET    /api/v1/search/{id}/status  # 查询进度
GET    /api/v1/search/{id}/result  # 获取结果
```

| 关注点 | 说明 |
|--------|------|
| `_results_cache` | 内存字典缓存结果（生产需换 Redis） |
| `BackgroundTasks` | FastAPI 的后台任务机制 |
| `submit-then-poll` 模式 | 适合长任务，但不够实时 |

---

### 5.3 `src/gateway/chat_router.py` — 多轮对话 API

```
POST   /api/v1/chat                     # SSE 流式对话
GET    /api/v1/chat/{id}/history        # 会话历史
GET    /api/v1/chat/{id}/summary        # 会话摘要
DELETE /api/v1/chat/{id}                # 结束会话
```

| 关注点 | 说明 |
|--------|------|
| `EventSourceResponse` | `sse-starlette` 提供的 SSE 响应类 |
| `asyncio.Queue` 桥接 | Agent 回调 → Queue → SSE yield |
| `event_generator()` | 异步生成器：session → progress* → message → result? → done |

**复现要点**：SSE 的核心在于 `asyncio.Queue` 解耦——Agent 处理线程推入进度，HTTP 流线程消费并 yield。两边的速度不同步，Queue 充当缓冲区。

---

### 5.4 `src/gateway/server.py`

| 关注点 | 说明 |
|--------|------|
| `lifespan` | FastAPI 生命周期：启动时 `init_db()`，关闭时 `close_db()` |
| `create_app()` | 应用工厂：创建 FastAPI → 注册中间件 → 注册路由 |

---

### 5.5 `src/main.py`

仅 3 行核心代码：`app = create_app()`，然后 `uvicorn.run()`。真正的逻辑都在 `server.py` 的工厂函数里。

## 复现路线图

如果你想从零手写这个项目，建议按以下顺序实现：

```
Phase 1: 骨架
  Day 1: config.py + logger.py + .env.example
  Day 2: schemas.py（所有数据模型）
  Day 3: base.py（Agent 基类 + ask_llm + ask_llm_json）

Phase 2: 单 Agent
  Day 4: scoring.py（纯规则引擎，可独立测试）
  Day 5: finder/prompts.py + finder/agent.py
  Day 6: encyclopedia/prompts.py + encyclopedia/scrapers.py + encyclopedia/agent.py
  Day 7: calculator/prompts.py + calculator/agent.py

Phase 3: 编排
  Day 8: mcp/tools.py + mcp/xianyu_server.py
  Day 9: orchestrator/agent.py + orchestrator/workflow.py
  Day 10: server.py + router.py + main.py（一次性搜索 API 可用了！）

Phase 4: 对话
  Day 11: session.py（会话管理）
  Day 12: chat/prompts.py + chat/agent.py
  Day 13: chat_router.py（SSE 流式）
  Day 14: 联调 + 修复

Phase 5: 部署
  Day 15: Dockerfile + docker-compose.yml + init_db.sql
```
